# Customer API

Samparka exposes partner customer APIs so RestroX can search, create, update, and upsert customers using store-scoped normalized phone identity.

## Purpose

Use this page to implement:

- customer search
- customer retrieval
- customer create
- customer update
- customer upsert

## Concepts

All customer partner routes require:

- partner authentication
- `x-integration-key`
- provider path parameter

The current implementation supports provider `restrox`.

## Authentication

Required headers:

```http
x-partner-key: your-partner-key
x-integration-key: SPK-RX-ABC12345
```

You can also send the partner key as:

```http
Authorization: Bearer your-partner-key
```

## Customer Identity Notes

These endpoints depend on the identity outcomes documented in [Customer Identity](./customer-identity):

- `found`
- `missing`
- `created`
- `conflict`
- `invalid_phone`

## Endpoints

### Search Customer

```http
GET /api/partners/{provider}/customers/search?phone=9801234567
```

Supported query:

- `phone`

No other search keys are accepted by the verified implementation.

Example success response when found:

```json
{
  "exists": true,
  "customer": {
    "id": "687000000000000000000001",
    "name": "Asha",
    "phone": "9801234567",
    "email": "asha@example.com",
    "points": 45,
    "tier": "Bronze Tier",
    "lifetimePoints": 45,
    "membershipSince": "2026-06-08T10:15:00.000Z"
  }
}
```

Example success response when missing:

```json
{
  "exists": false
}
```

Failure cases:

- `400 Phone is required`
- `400 Search supports exact phone lookup only`
- `400 Invalid phone number`
- `409 customer_identity_conflict`

### Get Customer

```http
GET /api/partners/{provider}/customers/{customerId}
```

Example response:

```json
{
  "customer": {
    "id": "687000000000000000000001",
    "name": "Asha",
    "phone": "9801234567",
    "email": "asha@example.com",
    "points": 45,
    "tier": "Bronze Tier",
    "lifetimePoints": 45,
    "membershipSince": "2026-06-08T10:15:00.000Z"
  }
}
```

Failure cases:

- `404 Customer not found`

### Create Customer

```http
POST /api/partners/{provider}/customers
Content-Type: application/json
```

Example request:

```json
{
  "name": "Asha",
  "phone": "+9779801234567",
  "email": "asha@example.com"
}
```

Behavior:

- if exactly one matching customer exists, the existing customer is returned
- if no matching customer exists, a new customer is created
- if multiple matching customers exist, the request fails with conflict

Example response:

```json
{
  "customer": {
    "id": "687000000000000000000001",
    "name": "Asha",
    "phone": "9801234567",
    "email": "asha@example.com",
    "points": 0,
    "tier": "Bronze Tier",
    "lifetimePoints": 0,
    "membershipSince": "2026-06-09T10:15:00.000Z"
  }
}
```

Validation rules:

- `name` required
- `phone` required
- `email` optional but must be valid if present

Failure cases:

- `400 Name is required`
- `400 Phone is required`
- `400 Invalid phone number`
- `400 Invalid email`
- `409 customer_identity_conflict`
- `409 Email already exists`

### Update Customer

```http
PATCH /api/partners/{provider}/customers/{customerId}
Content-Type: application/json
```

Updatable fields:

- `name`
- `email`

Example request:

```json
{
  "name": "Asha Sharma",
  "email": "asha.sharma@example.com"
}
```

Example response:

```json
{
  "customer": {
    "id": "687000000000000000000001",
    "name": "Asha Sharma",
    "phone": "9801234567",
    "email": "asha.sharma@example.com",
    "points": 45,
    "tier": "Bronze Tier",
    "lifetimePoints": 45,
    "membershipSince": "2026-06-08T10:15:00.000Z"
  }
}
```

Validation rules:

- at least one field required
- only `name` and `email` are allowed
- `email` must be valid if present

Failure cases:

- `400 At least one updatable field is required`
- `400 Only name and email can be updated`
- `400 Invalid email`
- `404 Customer not found`
- `409 Email already exists`

### Upsert Customer

```http
POST /api/partners/{provider}/customers/upsert
Content-Type: application/json
```

Example request:

```json
{
  "name": "Asha",
  "phone": "9779801234567",
  "email": "asha@example.com"
}
```

Behavior:

- if exactly one matching customer exists, return it with `created: false`
- if no match exists, create one and return `created: true`
- if multiple matches exist, fail with conflict

Example created response:

```json
{
  "created": true,
  "customer": {
    "id": "687000000000000000000001",
    "name": "Asha",
    "phone": "9801234567",
    "email": "asha@example.com",
    "points": 0,
    "tier": "Bronze Tier",
    "lifetimePoints": 0,
    "membershipSince": "2026-06-09T10:15:00.000Z"
  }
}
```

Example existing response:

```json
{
  "created": false,
  "customer": {
    "id": "687000000000000000000001",
    "name": "Asha",
    "phone": "9801234567",
    "email": "asha@example.com",
    "points": 45,
    "tier": "Bronze Tier",
    "lifetimePoints": 45,
    "membershipSince": "2026-06-08T10:15:00.000Z"
  }
}
```

Failure cases:

- `400 Phone is required`
- `400 Name is required` when a new customer would need to be created
- `400 Invalid phone number`
- `400 Invalid email`
- `409 customer_identity_conflict`
- `409 Email already exists`

## Duplicate Prevention Notes

Duplicate prevention is store-scoped and phone-first.

- create returns existing customer when one normalized-phone match exists
- upsert returns existing customer with `created: false`
- conflict blocks create/upsert when more than one normalized-phone match exists

## Idempotency Notes

The partner customer endpoints do not expose a separate idempotency key header in the verified implementation.

Current practical behavior:

- repeated create calls for the same normalized phone return the existing customer once one match exists
- repeated upsert calls for the same normalized phone are safe at the application level

## Operational Notes

- Partner customer auth requires an active integration.
- New partner-created customers use `customer_source = "pos"` and provider metadata for RestroX.
- Customer serialization exposes `id`, `name`, `phone`, `email`, `points`, `tier`, `lifetimePoints`, and `membershipSince`.

## Troubleshooting Notes

- If search fails, check the phone format first.
- If update fails, verify you are only sending `name` and `email`.
- If create or upsert fails with conflict, manual customer cleanup is required.
