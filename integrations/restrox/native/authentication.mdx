RestroX uses three different credentials depending on the surface.

## Credential Matrix

| Surface | Credential | Notes |
| ------ | ------- | ----- |
| Merchant integration APIs | Merchant-authenticated session or bearer token | Used to create, inspect, disconnect, and manage the integration. |
| Partner APIs | Partner key + Integration Key in request body or header | `x-partner-key` authenticates the partner. `integrationKey` resolves the outlet-owned integration. |
| Customer APIs | Partner key + `x-integration-key` | Customer routes are read-only and store-scoped. |
| Webhook delivery | Webhook token in URL path | `POST /webhook/restrox/{token}` |

## Webhook Authentication

Webhook delivery uses the integration-level token in the URL path. RestroX does not need a separate signature header for the currently documented contract.

## Customer API Failure Cases

- `503 RestroX partner API is not configured`
- `401` for invalid partner or integration authentication
- `403` when the integration is disconnected
