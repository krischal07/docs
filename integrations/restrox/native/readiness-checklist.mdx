# Readiness Checklist

Use this checklist to confirm the RestroX native integration is ready for production traffic.

## Purpose

This page turns the verified native onboarding flow into an implementation checklist for RestroX engineers.

## Checklist

- [ ] Connection Key validated
- [ ] Merchant connected
- [ ] Locations synced
- [ ] Mappings reviewed
- [ ] Customer lookup verified
- [ ] Customer upsert verified
- [ ] Test sale completed
- [ ] Loyalty awarded
- [ ] Refund tested
- [ ] Webhook delivery verified
- [ ] Ready for production

## Workflow

<Steps>
  <Step title="Validate Connection Key">
    Confirm the active Connection Key works through the native partner connect flow.
  </Step>
  <Step title="Connect Merchant">
    Call the RestroX partner connect route and confirm the merchant enters a connected or reviewable state.
  </Step>
  <Step title="Sync And Review Locations">
    Sync locations, resolve review issues, and confirm no required location is blocked.
  </Step>
  <Step title="Verify Customer Operations">
    Test customer search and upsert using normalized phone input.
  </Step>
  <Step title="Submit Test Sale">
    Send a completed sale through the native test-sale path and confirm it enters the canonical token-routed ingress successfully.
  </Step>
  <Step title="Validate Readiness">
    Confirm the integration progresses into a ready state and optionally verify refund handling.
  </Step>
</Steps>

## Operational Notes

- `Needs Review` should be resolved before relying on production traffic.
- Refund verification is optional in the current merchant status builder, but it is still recommended before go-live.
- Test-sale validates the same token-routed webhook ingress used by production webhook traffic.

## Troubleshooting Notes

- If the test sale is accepted but readiness does not advance, check location resolution, location state, and customer assumptions.
- If customer operations fail, resolve identity or integration-key issues before retrying sales.
