Samparka exposes three RestroX partner routes:

- `POST /api/partners/restrox/connect`
- `POST /api/partners/restrox/sync-locations`
- `POST /api/partners/restrox/test-sale`

## Connect

Use the Integration Key plus a restaurant identifier. The service stores the binding on `PosIntegration.external_location_id` and `external_location_name`.

```json
{
  "integrationKey": "SPK-RX-ABC12345",
  "account": {
    "id": "restrox-account-001",
    "name": "Java Express"
  },
  "locations": [
    {
      "restaurantId": "12345",
      "restaurantName": "Kathmandu Branch"
    }
  ]
}
```

Successful connect sets the integration lifecycle status to `CONNECTED`.

## Skipped Sync Route

The route still exists, but for outlet-owned RestroX it returns a skipped no-op response:

```json
{
  "connected": true,
  "skipped": true,
  "message": "Outlet-owned RestroX integrations skip this shared route."
}
```

## Test Sale

`test-sale` reuses the stored restaurant binding and then forwards the event into the normal webhook pipeline.

```json
{
  "integrationKey": "SPK-RX-ABC12345",
  "payload": {
    "event_type": "order.completed",
    "order_id": "restrox-sale-1001",
    "restaurantId": "12345",
    "amount": 850
  }
}
```

If the payload restaurant identifier disagrees with the stored binding, the route returns `409`.
