# Error Codes

The current RestroX native integration exposes partner-visible error messages, but it does not consistently expose a stable symbolic error code for every route family.

## Purpose

Use this page to understand:

- what partner-visible errors are currently observable
- which surfaces return symbolic error codes
- which gaps still require clarification

## Current Behavior

### Partner Routes

RestroX partner routes use:

```json
{
  "success": false,
  "message": "..."
}
```

### Partner Customer Routes

Partner customer routes use:

```json
{
  "error": "partner_customer_error",
  "message": "..."
}
```

or, for customer identity conflict:

```json
{
  "error": "customer_identity_conflict",
  "message": "Multiple customer records share the same normalized phone. Admin resolution is required."
}
```

## Requested Symbolic Codes

The following symbolic labels are useful for integrators, but they are not all implemented as stable partner-facing codes today:

- `INVALID_CONNECTION_KEY`
- `PARTNER_NOT_CONNECTED`
- `LOCATION_NOT_MAPPED`
- `CUSTOMER_CONFLICT`
- `INVALID_PHONE`
- `CUSTOMER_NOT_FOUND`
- `INTEGRATION_NOT_READY`

Current mapping:

- `CUSTOMER_CONFLICT` is partially represented by `customer_identity_conflict`
- `INVALID_PHONE` is currently exposed through message text only
- `CUSTOMER_NOT_FOUND` is currently exposed through message text only

Implementation detail requires clarification.

The current partner/public routes do not publish a complete stable symbolic error catalog for all native integration scenarios.

## Failure Scenarios

### Invalid Connection Key

Observed as:

```json
{
  "success": false,
  "message": "Invalid Integration Key"
}
```

### Customer Conflict

Observed as:

```json
{
  "error": "customer_identity_conflict",
  "message": "Multiple customer records share the same normalized phone. Admin resolution is required."
}
```

### Invalid Phone

Observed as:

```json
{
  "error": "partner_customer_error",
  "message": "Invalid phone number"
}
```

## Operational Notes

- Route families do not currently share one unified symbolic error format.
- If you need stable machine-level error classification, use HTTP status plus message and the available partner customer `error` field.

## Troubleshooting Notes

- If you rely on symbolic error codes, build defensive handling around current messages until a formal cross-route error catalog is published.
- Distinguish partner route failures from partner customer failures because their response shapes differ.
